# 银行IT架构管理平台 — 开发计划

**项目名称**：银行IT架构管理平台
**版本**：V1.0（一期）
**日期**：2026-03-26
**状态**：待审核

---

## 一、项目概述

### 1.1 项目目标

建设一套覆盖全行的IT架构管理平台，实现：
- 技术基础设施（CMDB）统一管理
- 应用架构、技术架构、数据架构三维可视化管理
- 新建/变更应用的多级审批流程
- 面向管理层和审计的报表与全景图

### 1.2 一期范围

| 模块 | 功能 |
|------|------|
| CMDB管理 | 服务器、网络、存储、云资源的配置管理，支持手工录入和文件导入 |
| 应用架构 | 应用系统、服务架构、模块划分、依赖关系 |
| 技术架构 | 技术栈、部署架构、中间件和数据库分布 |
| 数据架构 | 数据分布、数据流向、数据实体关系 |
| 审批流程 | 多级审批，支持流程配置 |
| 权限管理 | 角色权限控制，对接行内身份认证 |
| 报表中心 | 统计报表、三全景图、合规报告、趋势分析 |

### 1.3 二期范围（本期不含）

- 监控告警
- 变更全流程管理

## 二、技术方案

### 2.1 技术栈

| 层级 | 技术选型 |
|------|----------|
| 前端 | Vue.js 3 + Element Plus |
| 后端 | Spring Boot 3 + MyBatis-Plus |
| 数据库 | MySQL 8.0 / PostgreSQL |
| 缓存 | Redis |
| 文件存储 | 本地存储（可扩展至对象存储） |
| 部署 | Docker / 行内云平台 |

### 2.2 系统架构

```
┌─────────────────────────────────────────────────────────────┐
│                      前端 (Vue.js)                           │
│    门户首页 | CMDB | 应用架构 | 技术架构 | 数据架构 | 审批 | 报表  │
└──────────────────────────┬──────────────────────────────────┘
                           │ REST API / WebSocket
┌──────────────────────────▼──────────────────────────────────┐
│                    后端( Spring Boot )                        │
│  ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐     │
│  │ CMDB   │ │ 架构   │ │ 审批   │ │ 报表   │ │ 权限   │     │
│  │ 服务   │ │ 服务   │ │ 服务   │ │ 服务   │ │ 服务   │     │
│  └────────┘ └────────┘ └────────┘ └────────┘ └────────┘     │
│  ┌────────┐ ┌────────┐                                        │
│  │ 用户   │ │ 文件   │                                        │
│  │ 服务   │ │ 服务   │                                        │
│  └────────┘ └────────┘                                        │
└──────────────────────────┬──────────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────────┐
│                    MySQL / PostgreSQL                         │
│    CMDB库 | 架构库 | 审批库 | 报表库 | 权限库                   │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 部署架构

- **开发环境**：Docker Compose一键部署
- **测试/生产环境**：行内云平台（K8s或行内私有云）

## 四、功能详细设计

### 4.1 CMDB管理

**功能清单**：
- 服务器/网络/存储/云资源的增删改查
- 批量导入（Excel模板）
- 批量导出
- 资产分类检索
- 资产变更记录

**数据字段（服务器）**：
| 字段 | 说明 |
|------|------|
| hostname | 主机名 |
| ip | IP地址 |
| cpu | CPU规格 |
| memory | 内存规格 |
| disk | 磁盘规格 |
| os | 操作系统 |
| status | 运行状态 |
| department | 所属部门 |
| location | 机房位置 |

### 4.2 应用架构

**功能清单**：
- 应用系统列表（支持树形/列表视图）
- 应用详情（基本信息、模块列表、部署信息）
- 服务划分
- 依赖关系管理（依赖/被依赖）
- 应用搜索和筛选

### 4.3 技术架构

**功能清单**：
- 技术栈配置（语言、框架、中间件、数据库）
- 部署架构图
- 技术栈分布统计
- 技术选型规范（可配置标准技术栈）

### 4.4 数据架构

**功能清单**：
- 数据实体管理
- 数据流向设计
- 数据分布视图
- 数据敏感等级标注

### 4.5 审批流程

**功能清单**：
- 流程定义（多级审批链）
- 新建应用申请
- 架构变更申请
- 审批任务（待办/已办）
- 审批历史追溯
- 流程状态跟踪

**默认审批流程**：
```
申请人提交 → 部门负责人审批 → 科技管理层审批 → 完成
            （如为核心系统，增设架构委员会审批）
```

### 4.6 权限管理

**功能清单**：
- 用户管理（对接行内AD/LDAP）
- 角色配置（超级管理员/架构师/开发/项目经理/审计等）
- 权限配置（菜单/按钮/数据级别）
- 权限变更记录

### 4.7 报表中心

**功能清单**：
| 报表 | 说明 |
|------|------|
| 系统统计 | 按部门/状态/生命周期统计 |
| 技术分布 | 技术栈使用排行 |
| 三全景图 | 系统/技术栈/数据分布可视化 |
| 合规报告 | 变更记录导出、审计报告 |
| 趋势分析 | 架构演进历史 |

## 六、开发阶段

### 第一阶段：基础设施（4周）

**目标**：完成项目脚手架、后端框架、前端框架、数据库设计

**任务**：
1. 项目工程创建（前端Vue项目 + 后端Spring Boot项目）
2. 数据库建模（PowerDesigner或PDMan设计）
3. 后端：Spring Boot框架搭建（认证、日志、异常处理）
4. 前端：Vue3 + Element Plus框架搭建（路由、状态管理）
5. 代码仓库初始化，Git Flow流程配置
6. 接口规范制定（REST API规范）

**交付物**：
- 项目脚手架代码
- 数据库DDL脚本
- 接口设计文档

---

### 第二阶段：CMDB模块（4周）

**目标**：完成CMDB全部功能

**任务**：
1. 后端CMDB服务开发（CRUD、批量操作）
2. 前端CMDB页面开发（列表、表单、详情）
3. 文件导入导出功能（Excel模板、解析、导出）
4. 资产变更记录
5. CMDB统计面板

**交付物**：
- CMDB功能代码
- 单元测试

---

### 第三阶段：架构管理模块（5周）

**目标**：完成应用架构、技术架构、数据架构三个维度

**任务**：

**3.1 应用架构（2周）**
- 应用系统CRUD
- 服务和模块管理
- 依赖关系管理
- 依赖关系可视化

**3.2 技术架构（1.5周）**
- 技术栈管理
- 部署架构视图
- 技术分布统计

**3.3 数据架构（1.5周）**
- 数据实体管理
- 数据流向设计
- 数据分布视图

**交付物**：
- 架构管理功能代码
- 单元测试

---

### 第四阶段：权限与审批（4周）

**目标**：完成权限管理和多级审批

**任务**：

**4.1 权限管理（1.5周）**
- 用户管理（对接LDAP/AD）
- 角色管理
- 权限配置（菜单/按钮/数据级别）
- 权限变更记录

**4.2 审批流程（2.5周）**
- 流程定义引擎（多级审批链）
- 审批任务（待办/已办/历史）
- 审批通知（站内信）
- 审批历史追溯

**交付物**：
- 权限和审批功能代码
- 单元测试

---

## 九、风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| 需求变更频繁 | 进度延迟 | 需求冻结后再进入开发阶段；预留缓冲时间 |
| LDAP/AD对接复杂 | 影响权限模块进度 | 前期充分沟通接口；二期再做对接 |
| 依赖关系可视化性能 | 大规模系统展示慢 | 考虑虚拟化渲染；分页加载 |
| 数据迁移 | 历史数据丢失风险 | 制定详细迁移方案；多次演练 |
| 用户培训不足 | 上线后使用率低 | 充分培训；上线后持续支持 |
