# 登录与菜单权限管理功能设计方案

**版本**：V1.0
**日期**：2026-03-27
**状态**：待审核

---

## 一、背景与目标

### 1.1 现状问题

当前系统缺乏认证机制，所有页面无需登录即可访问，存在以下问题：
- 无用户身份验证，任何人可直接访问系统
- 无权限控制，所有用户可见全部功能
- 无会话管理，无法追踪用户操作

### 1.2 目标

1. **登录认证** - 专业登录页面 + JWT会话机制
2. **权限控制** - 基于角色的菜单和按钮级别权限控制
3. **菜单管理** - 树状菜单管理（仅超级管理员可访问）
4. **角色权限配置** - 角色关联菜单/按钮权限

---

## 二、技术方案

### 2.1 技术选型

| 组件 | 技术选型 | 说明 |
|------|----------|------|
| 会话管理 | JWT (jjwt 0.12.3) | Token存储在localStorage，后端不维护会话状态 |
| 登录页面 | Vue 3 + Element Plus | 使用渐变背景+卡片式设计 |
| 权限控制 | 后端拦截器 + 前端路由守卫 | 双层防护 |
| 密码加密 | MD5 (Hutool SecureUtil) | 与现有系统保持一致 |

### 2.2 设计原则

- **菜单和按钮存储在同一张表（sys_menu）**，通过 `menu_type` 区分（MENU/BUTTON）
- **按钮作为菜单的子级**，通过 `parent_id` 关联到父菜单
- **角色菜单权限表（sys_role_menu）** 直接关联 `menu_id`，即可授权菜单也可授权按钮
- **层级约束**：授权按钮权限时，必须先拥有其父菜单权限

## 三、数据库设计

### 3.1 新增表

#### sys_user_role (用户角色关联表)

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键 |
| user_id | BIGINT | 用户ID |
| role_id | BIGINT | 角色ID |
| create_time | DATETIME | 创建时间 |

```sql
CREATE TABLE sys_user_role (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT NOT NULL COMMENT '用户ID',
    role_id BIGINT NOT NULL COMMENT '角色ID',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_user_role (user_id, role_id),
    INDEX idx_user_id (user_id),
    INDEX idx_role_id (role_id)
) COMMENT '用户角色关联表';
```

#### sys_role_menu (角色菜单权限表)

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键 |
| role_id | BIGINT | 角色ID |
| menu_id | BIGINT | 菜单ID（可以是菜单或按钮） |
| create_time | DATETIME | 创建时间 |

```sql
CREATE TABLE sys_role_menu (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    role_id BIGINT NOT NULL COMMENT '角色ID',
    menu_id BIGINT NOT NULL COMMENT '菜单ID（可以是菜单或按钮）',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_role_menu (role_id, menu_id),
    INDEX idx_role_id (role_id),
    INDEX idx_menu_id (menu_id)
) COMMENT '角色菜单权限表';
```

### 4.3 JWT认证流程

```
1. 用户登录 → POST /v1/auth/login
2. 验证用户密码（MD5）→ 匹配成功生成JWT Token
3. Token包含: userId, username, roleIds, expiration
4. 前端存储Token到localStorage
5. 后续请求Header携带: Authorization: Bearer <token>
6. JwtInterceptor拦截所有请求（除/login），验证Token
7. 从Token解析用户信息，设置到ThreadLocal
8. 验证通过则放行，失败返回401
```

### 5.3 路由守卫逻辑

```javascript
router.beforeEach((to, from, next) => {
  const token = localStorage.getItem('token')
  if (to.path === '/login') {
    token ? next('/') : next()
  } else {
    token ? next() : next('/login')
  }
})
```

### 5.4 动态菜单渲染

App.vue 根据用户菜单权限动态渲染侧边栏：
- 从 authStore 获取用户菜单权限树
- 只显示用户有权限访问的菜单项
- 无权限的菜单不显示

### 5.5 按钮权限控制

```javascript
// v-permission 指令
app.directive('permission', {
  mounted(el, binding) {
    if (!authStore.hasPermission(binding.value)) {
      el.parentNode.removeChild(el)
    }
  }
})

// 使用方式
<el-button v-permission="'cmdb:server:add'">新增</el-button>
```

## 九、风险与注意事项

1. **密码兼容性**：现有数据库中的admin用户密码为BCrypt格式，需统一改为MD5
2. **会话超时**：JWT Token设置24小时过期，前端需处理过期跳转登录
3. **并发登录**：同一用户多处登录会生成多个Token，需考虑是否限制单点登录
4. **权限缓存**：用户权限可在前端缓存，减少频繁查询
